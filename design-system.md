# Дизайн-система

Это канонический ориентир для активных экранов приложения.

Цель системы:
- держать интерфейс спокойным и взрослым;
- не расползаться в новые цвета и случайные типографические решения;
- поддерживать ощущение "живого", но не шумного продукта.

## Характер

Интерфейс должен ощущаться так:
- светлый;
- тёплый;
- собранный;
- поддерживающий;
- без визуальной тревоги;
- без "дешёвой" геймификации.

Ключевой принцип:
- один главный смысл на экран;
- один главный акцент в карточке;
- вторичный слой должен поддерживать, а не спорить.

## Цвета

### Базовая палитра

- `#1e1c18` — Основной
- `#1D9E75` — Успех / опора
- `#EF9F27` — Внимание
- `#D85A30` — Риск
- `#faf9f6` — Фон поверхности
- `#ece7dd` — Внешний тёплый фон
- `#ffffff` — Поднятая поверхность
- `#f3efe7` — Мягкая подложка
- `#e6ded2` — Тонкая граница

### Роли цветов

`Основной`
- заголовки;
- основной текст;
- тёмные action-блоки;
- активные tab-состояния.

`Успех / опора`
- живой фокус;
- поддержка;
- спокойные positive-сигналы;
- рабочие опоры;
- удержание;
- progress / healthy state.

`Внимание`
- важное окно;
- умеренное напряжение;
- "стоит заметить";
- промежуточный статус;
- не использовать как цвет ошибки.

`Риск`
- высокий риск;
- перегруз;
- интенсивная тяга;
- тяжёлые паттерны;
- только для действительно напряжённых вещей.

`Фон`
- чистый лист;
- спокойные карточки;
- отсутствие медицинской или тревожной эстетики.

### Что не делать

- не добавлять новые акцентные цвета без необходимости;
- не красить один экран в 5 смыслов сразу;
- не использовать риск-оранжевый как обычный CTA;
- не использовать зелёный как "молодец" на каждом шаге.

## Типографика

Базовая гарнитура:
- `Manrope`

Сейчас это наша единая система.
Не вводим вторую гарнитуру без сильной причины.

### Уровни текста

`Заголовок`
- крупный;
- уверенный;
- короткий;
- плотный по трекингу;
- используется для ключевого смысла экрана.

Примеры:
- `Доброе утро`
- `Что произошло?`
- `Инсайты`

`Подзаголовок`
- спокойный;
- объясняет контекст;
- не длинный;
- не спорит с заголовком.

Примеры:
- `Ориентир сегодня`
- `Что стоит за паттерном`

`Тело`
- основной читаемый текст;
- line-height свободнее;
- не серый и не слишком мелкий.

Пример:
- `Каждая запись — честность с собой.`

`Подпись`
- очень тихий слой;
- только служебная информация;
- не привлекать к себе внимание.

Пример:
- `из 12 дней наблюдений`

### Технические токены

Используемые значения в активных экранах:
- `--font-display: var(--font-sans)`
- `--tracking-display: -0.03em` или `-0.035em`
- `--tracking-tight: -0.02em`
- `--tracking-caps: 0.08em`–`0.1em`

### Типографические правила

- не делать много жирного подряд;
- uppercase — только для маленьких служебных лейблов;
- большие числа и ключевые заголовки могут быть чуть плотнее по трекингу;
- подписи не должны конкурировать с телом текста;
- если текст можно сделать короче — делать короче.

## Карточки

Наша карточка — это:
- светлая поверхность;
- мягкое скругление;
- тонкая граница;
- лёгкая тень;
- один главный смысл.

Хорошая карточка:
- объясняет один кусок;
- не требует "расшифровывать" интерфейс;
- не выглядит как техническая панель.

Плохая карточка:
- содержит 3 равноправных мысли;
- кричит цветом;
- делает пользователя "студентом отчёта".

## Акценты

Используем 3 типа акцента:

`Вертикальный акцент`
- для главного наблюдения;
- для ориентира дня;
- для связанных контекстных блоков.

`Мягкий бейдж`
- для статуса периода;
- для worked/sometimes;
- для спокойных меток.

`Тёмный action-блок`
- для важного действия;
- не более 1–2 на экран.

## Экранные принципы

### Главная

Должна отвечать на вопрос:
- `что мне важно прямо сейчас?`

Первый слой:
- момент дня;
- ритм дня;
- быстрое действие;
- короткая личная опора.

### Дневник

Должен отвечать на вопрос:
- `что произошло и как это связано с моим днём?`

Принципы:
- свободный ввод;
- мягкие теги;
- равноправие `удержался` и `сорвался`;
- связь записи с ориентиром дня.

### Инсайты

Должны отвечать на вопрос:
- `что я начинаю про себя понимать?`

Первый слой:
- одно главное наблюдение;
- карта периода;
- триггеры;
- что работало;
- паттерн риска.

### Профиль

Должен отвечать на вопрос:
- `какая у меня сейчас база и что помогает приложению быть точнее?`

Принципы:
- не перегружать всплывашками;
- важный контекст открывать отдельным focused screen;
- AI — только opt-in и с предупреждением.

## Язык интерфейса

Наш язык:
- тёплый;
- ясный;
- без осуждения;
- без псевдо-медицинской тяжести;
- без роли терапевта.

Формулировки должны быть:
- человеческими;
- короткими;
- точными;
- персональными, когда это возможно.

Не писать:
- сухо;
- канцелярски;
- как диагноз;
- как отчёт для врача.

## Практический фильтр

Перед добавлением нового блока проверяем:

1. Это действительно помогает сейчас?
2. Это не дублирует уже существующий смысл?
3. Это не создаёт когнитивную длину?
4. Это звучит как поддержка, а не как контроль?
5. Это визуально живёт в нашей палитре и иерархии?

Если хотя бы на два пункта ответ "нет", блок не добавляем в первый слой.
